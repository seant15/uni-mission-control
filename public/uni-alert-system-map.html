🔴Layer 0/1 跨日重复触发

VPS 每天重新触发同一问题，无跨日去重。zero_spend + under_pacing 可在同一账户同时触发（是同一件事）。7天 339 条，实际独立问题约 35 个。

建议：在 VPS 评估脚本加 resolved_at IS NULL 检查，已有未解决的同类警报就不重发。

⚠️Meta CTR 口径变更 2026-03

Meta 2026年3月起 CTR = 链接点击数（不再包含所有点击）。基准里的历史 CTR 数据包含旧口径，导致 ctr_decline 可能误报。

建议：Meta CTR 基准只使用 2026-03-01 之后的数据计算。已在 update_baselines.py 中标记 baseline_cutoff_date。

⚠️PROD 账户疑似测试账户

PROD 客户日均预期 $992，连续 7 天 $0 花费，Google + Meta 同时为零。如果是测试账户应从警报中排除，否则这是一个真实的严重问题。

建议：在 clients 表加 is_active 字段，或人工确认 PROD 状态。

⚠️3 个客户无 Baselines

18 个客户中 3 个没有 account_daily_baselines 数据，导致所有 Layer 2+ 警报对这些客户无效。

建议：检查这 3 个客户的 daily_performance 数据是否正常同步，数据少于 7 天则 Layer 2 不触发。

ℹ️转化延迟 Window

Layer 2 所有检查都用 ≥48h 前的数据，规避 Meta 24-72h 转化回传延迟。这意味着最新问题最快 48h 后才被 Layer 2 捕获，Layer 0/1 负责实时补位。

这是设计行为，不是问题。

ℹ️无季节性抑制

黑五、重大节假日、主动暂停投放期间，系统仍会触发 zero_spend 和 under_pacing。目前只能手动 resolve/ignore。

建议：增加 suppression_windows 表，让账户经理提前标记已知的暂停窗口。

建设建议 — 辅助人工 Media Buying
REC-01
警报 → 行动 标准SOP映射
每种 alert_type 关联一个固定 SOP（排查步骤 + 负责人 + 响应时限）。Account manager 看到警报就知道该做什么，不用从头分析。比如 landing_page_failure → 检查 PageSpeed + 表单 → 通知 Web 团队 → 4h内响应。
影响：减少响应时间 60%
REC-02
Auto-Resolve 自动关闭逻辑
如果昨日触发 under_pacing，但今日花费已恢复正常，自动将该警报标记为 resolved。目前警报只会堆积不会自动清零，导致 Dashboard 噪音极大，AM 容易产生"警报疲劳"。
影响：噪音减少 ~80%
REC-03
客户级别阈值自定义
不同客户风险容忍度不同。预算小的客户 CPA 波动天然更大。在 clients 表加 roas_threshold_override、cpa_threshold_override 字段，让 AM 针对每个客户调整触发敏感度。
影响：减少误报
REC-04
Weekly Digest 周报摘要
每周一自动生成一份"上周警报摘要"——哪个客户警报最多、哪类问题最频繁、哪些连续3周都在触发。帮助团队识别慢性病（不是应急处理），推动策略层面的优化决策。
影响：发现慢性问题
REC-05
Suppression Windows 暂停窗口
增加 suppression_windows 表（client_id, start_date, end_date, reason）。AM 提前标记"这个客户本周暂停投放做落地页改版"，系统在这段时间内自动抑制该客户的 zero_spend 和 under_pacing 警报。
影响：消除已知场景噪音
REC-06
cpc_spike / creative_fatigue 优先
这两个是 Media Buyer 最高频手动检查的问题，但目前完全没有自动化。需要在 VPS sync 脚本里加 cpc 和 Meta frequency 字段的拉取，数据到位后 Layer 2 可以立刻添加这两个检查，对日常工作帮助最大。
影响：减少人工巡检
